ADAPTIVE LEARNING FEATURES REQUIRED

AYA Adaptive Learning Brain should do the following:

1. Weakness detection
AYA should detect weak topics from:
- repeated wrong answers
- repeated retries
- unfinished lessons
- low quiz accuracy
- frequent hint use

Examples:
- multiplication
- reading comprehension
- noun/adjective recognition
- fractions
- English vocabulary

2. Strength detection
AYA should detect strong areas from:
- repeated correct answers
- fast completion
- high quiz performance
- lesson streaks

3. Difficulty adaptation
If student struggles:
- easier explanations
- more step-by-step guidance
- more examples
- easier follow-up task

If student succeeds consistently:
- slightly harder tasks
- fewer hints
- faster progression

4. Lesson recommendation adaptation
AYA should prioritize:
- weak topics first
- review before new hard content
- reinforcement in Daily Plan when needed

5. Support adaptation
When student struggles:
- more encouragement
- smaller steps
- hints before full answer
- repetition of the concept in simpler words

6. Progress memory
AYA should remember learning state across sessions.

Examples:
- “student struggles with division”
- “student is strong in addition”
- “student needs more reading comprehension practice”

7. No punishment logic
AYA must never feel harsh.
Adaptive behavior must stay:
- supportive
- motivating
- child-friendly

--------------------------------
DATA / ARCHITECTURE RULES

1. Reuse current systems where possible
Integrate with existing:
- lesson engine
- daily plan
- rewards / XP
- chat
- progress tracking
- profile / student state

2. No duplicate tracking systems
Do NOT create:
- a second mastery engine
- a second progress engine
- a parallel weak-topic database

3. Prefer extension over duplication
If existing structures already store useful learning data, extend them instead of replacing them.

4. Keep the system lightweight
Do not add heavy over-engineering.
Use practical, maintainable logic.
